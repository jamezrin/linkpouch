# Incident Report — 2026-02-27

## Summary

A `kubectl apply -f k8s/ --recursive` run during a routine prod deployment
overwrote four live Kubernetes secrets with placeholder values from the git
repository, causing a cascade of failures: the site returned 502, metadata
indexing stopped, and screenshots stopped being stored.

---

Total downtime: **~8 minutes** (502 window: 16:21–16:29).

## Root Cause

The secret manifests in `k8s/` are committed as **templates** with
`CHANGE_ME_USE_EXTERNAL_SECRETS` placeholder values. They were introduced as
part of a security hardening commit that deliberately removed real credentials
from git. However, nothing prevents them from being applied with
`kubectl apply`, and when they are, they silently overwrite whatever real
values are live on the cluster.

A blanket `--recursive` apply hit all four secrets at once:

| Secret | Effect |
|---|---|
| `redis-secret` | Created fresh with placeholder → Redis suddenly required a password that no consumer knew |
| `postgres-secret` | Overwritten → stash-service DB password wrong |
| `stash-service-secret` | Overwritten → `SIGNATURE_MASTER_KEY` and `INDEXER_CALLBACK_SECRET` lost |
| `seaweedfs-secret` | Overwritten with new split-identity config → indexer S3 access key rejected |

A secondary issue was also revealed: the `network-policies.yaml`, applied for
the first time in prod, assumed Traefik lives in `kube-system`. On this
cluster Traefik runs in its own `traefik` namespace, so the ingress rules
blocked all external traffic immediately.

---

## Fixes Applied

1. **Redis** — removed `--requirepass` from the Redis deployment. Neither
   stash-service nor indexer-service pass `REDIS_PASSWORD`, so password-protected
   Redis was never going to work without further wiring. Reverted to
   unauthenticated Redis for now.

2. **postgres-secret** — patched back to `linkpouch-dev-secret-2024`, recovered
   from git history (commit `e4d22a6`, before credentials were scrubbed).

3. **seaweedfs-secret** — patched back to the original single-identity config
   (`linkpouch-access-key` / `linkpouch-secret-key-change-in-production`) that
   matches what the live SeaweedFS pod has loaded. The split-identity config
   introduced in the security hardening commit was never actually deployed.

4. **NetworkPolicies** — added `kubernetes.io/metadata.name: traefik` as an
   allowed ingress source in the `frontend` and `api-gateway` policies, alongside
   the existing `kube-system` rule.

5. **`docs/KUBERNETES_DEPLOYMENT.md`** — added a prominent warning against
   applying secret manifests with `kubectl apply`, with instructions for setting
   secrets manually on first deploy.

---

## Follow-up Actions

- **Adopt Sealed Secrets** (or SOPS / External Secrets Operator). Encrypted
  secrets can be safely committed and applied like any other manifest, eliminating
  the entire class of "placeholder clobbers live secret" accidents. This is the
  most important follow-up. See `docs/TODOS.md`.

- **Wire up Redis authentication properly**. The security hardening commit added
  `--requirepass` to Redis but never added `REDIS_PASSWORD` to the stash-service
  or indexer-service deployments. Redis is currently running without a password
  as a workaround.

- **Deploy the split SeaweedFS identity**. The security hardening commit intended
  to give the indexer write-only access and stash-service read/delete access.
  This was never applied because seaweedfs-secret was reverted. Should be
  done properly once Sealed Secrets is in place.

- **Never run `kubectl apply --recursive` on the full `k8s/` tree** until
  Sealed Secrets is adopted. See `docs/KUBERNETES_DEPLOYMENT.md` for the safe
  apply procedure.
